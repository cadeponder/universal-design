WCAG 2.0 Compliance checklist

The Web Content Accessibility Guidelines (WCAG 2.0) provide a checklist for developers to follow to ensure accessibility. View that checklist HERE.

WCAG 2.0 is organized under four foundational principles, which conveniently form the acronym POUR:

  • Perceivable: Information and user interface components must be presented to users in ways they can see or hear
  • Operable: User interface components and navigation must be designed so that users can interact with them and they can support assistive technologies such as screen readers
  • Understandable: Information and the operation of user interface must communicate clearly and consistently so that the content is readable
  • Robust: Content must be written so that it can be interpreted reliably by a wide variety of user agents, including assistive technologies

Using HTML5 Semantics

Allows accessibility technology to parse your website for folks with disabilities.

Some examples of HTML5 semantics:

  • <Header>
  • <Nav>
  • <Main>
  • <Article>
  • <Section>
  • <Aside>
  • <Footer>

Google Fonts

Google fonts is great for licensing considerations and ultimately not getting sued by font foundaries AND it allows you to search for fonts that are optimum for displaying content accessibly. An accessible font is one that is easy to see, read, and ultimately understand for all people

General Guidelines for Accessible Font Use:

  • Font sizes >=12pt is considered accessible but 16pt font is considered a safe font size for body copy
  • Keeping line length at <=100 characters per line
  • Serif fonts lend themselves to headers
  • Sans serif fonts are better for small to medium text along with headers
  • Avoiding "humanistic" fonts, such as scripted fonts, for larger bodies of text will help those with dyslexia or vision impairment
  • Guidelines may vary depending on purpose, length, etc. it is best to consult WCAG 2.0 for criteria when evaluating your choices
